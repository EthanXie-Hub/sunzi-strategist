# 仓库审查报告（2026-06-11）

审查对象：sunzi-strategist v0.2.0（SKILL.md + references/scorecards-and-gates.md + references/edge-lenses.md）

## 总判断

**这个 skill 已经是「现实战略诊断器」，不是「古文解释工具」。** 审查模板假设的大部分病（定位过虚、只讲文化、无追问、无反模式、无行动方案、输出空话）在 v0.2.0 已被解决，证据：

| 模板担心的问题 | 现状 | 证据位置 |
|---|---|---|
| 只解释古文、不解决问题 | 已是诊断器 | SKILL.md「使用原则」「五层主流程」 |
| 没有追问机制 | 有 | 原则 2「决定性未知 + 低成本侦察」 |
| 没有反模式判断 | 有 | 第五层（怒而兴师/五危/六败/君三患/费留） |
| 没有行动优先级 | 有 | 输出模板「接下来三步」「止损线」 |
| 容易输出战略大师式废话 | 有闸门 | 原则 5「框架要挣到位置」+「质量闸门」7 项自检 |
| 过度神化/阴谋化/攻击性 | 有护栏 | 原则 6（诡道≠欺骗信任你的人；烧关系的胜=破局）。已实测：「搞掉同事」题被重构为正当竞争 |
| 生硬套用兵法 | 有防线 | 原则 1（先判断是不是战略问题，简单题不套五层） |
| 不知道怎么输入 | 部分解决 | description 列了六类场景；但**没有 README**，GitHub 访客看不到 |
| 胜算评估太主观 | 有尺度 | scorecards-and-gates.md（10 分制+证据口径+补分动作） |

**真正缺的不是诊断能力，是开源工程外壳**：README、测试集、案例、反馈闭环、版本管理。本次重构补的就是这一层。

## 问题清单（按严重度）

### 严重（阻碍开源可用/可维护）
1. **无 git 仓库、无版本管理**——号称开源项目，实际是裸文件夹。改 6 轮无任何历史可回溯。→ 本次已 `git init` + 基线提交。
2. **无 README**——GitHub 访客（和未来的维护者）不知道这是什么、怎么装、怎么输入。→ 已新增。
3. **无评估集、无回归检查**——已手改 6+ 轮，没有任何固定测试防"改一处崩一处"（伦理护栏曾经就可能被静默删掉）。→ 已新增 evals/（20 用例 + 评分细则 + 回归清单）。
4. **知识底座未沉淀**——五类台账（框架/原则/技法/反模式/声音 + 原文锚点）只存在于开发对话历史里，仓库里没有。维护者无从判断"哪条是核心、哪条可省、原文出处在哪"。→ 已沉淀为 references/framework-ledger.md。

### 中等
5. **无反馈/复盘闭环**——skill 是一次性回答，好坏案例都流失。→ 已新增 feedback/（schema + AAR 七问 + 可控学习循环）。
6. **缺「对比决策」模式**——"A/B/C 方案选哪个"是高频输入（选 offer 已在 description 里），但输出模式 A–E 没有对应模板。→ 已新增模式 F。
7. **打包脚本会吞 .git**——skill-creator 的 package_skill.py 只排除 evals/、__pycache__ 等，**不排除 .git/**。git init 后再用它打包，.skill 会塞进整个 git 历史。→ REFACTOR_PLAN.md 给了安全打包命令；评估目录特意命名为 `evals/`（脚本自动排除）。
8. **无 LICENSE**——开源必需，但选哪个证照是所有者的法律决定。→ ✅ 已解决：所有者选定 MIT（仓库根 LICENSE），并复制到 `sunzi-strategist/LICENSE` 随 dist 包分发。

### 轻微
9. scorecards 里"对新玩家默认低分"等口径偏商业场景，用在职场/个人决策时需要模型自行换算——可接受，已在 ledger 标注。
10. 纯中文——限制国际受众。列入路线图（v0.4 可加英文 README），不阻塞当前。
11. 自动触发是概率性的（危险表面的题尤其会被模型"短路"绕过）——已有 /sunzi 显式命令缓解，属已知边界而非缺陷。

## 对审查模板建议的取舍（拒绝的部分及理由）

| 模板建议 | 取舍 | 理由 |
|---|---|---|
| knowledge/ 拆 6+ 文件（five-factors.md、seven-calculations.md…） | ❌ 拒绝，合并为一份 framework-ledger.md | 内容已在 SKILL.md 主流程里活用；再拆一套就是**双源漂移**——改一处忘另一处，开源维护的头号杀手 |
| 11 步固定诊断管道 | ❌ 拒绝 | 现有「6 原则 + 五层 + 质量闸门」已覆盖其 1–10 步，且保留了"简单题不套框架"的右尺寸能力；硬管道会把判断型任务框死。第 11 步（用户反馈）以轻量方式采纳 |
| Prompt 自动自我改进 | ❌ 拒绝自动，✅ 采纳人工闭环 | skill 无后端、不能也不该自改 Prompt；落地为 feedback/learning-loop.md：反馈→聚合→improvement-suggestions→人工 review→版本号合并→回归测试，防过拟合防污染 |
| 运行时日志/评分自动采集 | ❌ 拒绝 | skill 没有数据库；反馈以对话内邀请 + 用户自记 schema 文件实现 |
| 改名"孙子战略官" | ❌ 保留 sunzi-strategist | name 已被 description 召回测试（8/8）验证，改名重置一切 |
| 16 段输出结构 | ❌ 保留现有 A–F 模板 | 现有结构更紧且已实测；16 段是为"看起来全面"服务，违背"先给判断、能短则短" |
| 六大输出模板（含对比决策、风险自检） | ✅ 部分采纳 | 对比决策=新增模式 F；风险自检=已有模式 E（复盘/审计），不重复建 |
| 20 个测试用例、评分标准、README、CHANGELOG、反馈 schema、AAR | ✅ 全部采纳 | 正是真实缺口 |

## 真实使用障碍（普通用户视角）
- 装好后不知道说什么 → README 给了"直接把难题丢进来"的输入示例 + /sunzi 用法。
- 担心被说教/被教坏 → README 明示伦理边界（不教阴谋、不鼓励攻击，目标是减少误判、降低消耗、提高胜率）。
- 不知道输出长短怎么控 → README 说明精简/完整两档与切换话术。
