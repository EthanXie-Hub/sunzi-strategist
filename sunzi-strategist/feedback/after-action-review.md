# 复盘七问（After Action Review）

模式 E（复盘）的执行脚本。用户行动结束（或失败）回来时，按顺序走，**先还原后归因，不许跳到教训**。

1. **原目标**：当时要的结果是什么？（对照当时诊断的"什么算赢"）
2. **当时判断**：当时的庙算结论、主打法、止损线分别是什么？
3. **实际发生**：时间线还原，只摆事实不评价。
4. **判断对了哪条**：哪个透镜/推演被验证？（这要明确记下——对的也要归因，否则下次不敢再用）
5. **判断错了哪条**：哪个假设崩了？是当时就该看出来，还是事后才可能知道？
6. **归因四类**（必选其一为主因）：
   - **信息不足**——决定性未知没补就动了 → 改进侦察清单；
   - **框架误用**——透镜挂错或生搬 → 回写 ledger 的"何时用"；
   - **执行失败**——判断对但没做到位 → 不是 skill 的错，记执行教训；
   - **环境变化**——出现了当时无法预见的变量 → 检查"翻盘条件"当时有没有写出这一条。
7. **下次怎么改**：具体到"下次遇到 X 类问题，先问/先查/不做 Y"。写成一条可执行规则，进 feedback 记录。

输出按 1–7 编号给出，最后把第 6、7 条按 feedback-schema.md 落一条 JSON。

红线：复盘不是清算。涉及团队成员的失误，归因到机制与信息，不归因到人品——`数罚者，困也`，靠追责驱动的复盘以后再也收不到真话。
