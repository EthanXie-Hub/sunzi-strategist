# 可控学习循环

目标：让 skill 在持续使用中变准。原则：**学习发生在版本之间，不发生在运行之中**——skill 不自动改自己的 Prompt；一切改动可审查、可回滚、可版本化。

## 循环

```
使用 → 反馈(schema) / 复盘(AAR) 积累
  ↓  攒够一批（建议 ≥5 条，防过拟合）
聚合分析：哪个透镜常误用？哪类场景常缺问？哪些建议被证伪？
  ↓
生成 improvement-suggestions.md（用模板，附案例证据）
  ↓
人工 review：逐条 接受/拒绝/观察（拒绝也要写理由留档）
  ↓
改 SKILL.md / ledger / scorecards（最小改动，一次一主题）
  ↓
跑 evals/regression-checklist.md —— 不过不合并
  ↓
版本号 +0.0.1~0.1.0，CHANGELOG 记录，git commit
```

## 防腐三条

1. **防过拟合**：单一案例不改规则。一条建议至少要有 3 个独立案例支撑，或 1 个案例 + 明确的逻辑漏洞。一次失败可能只是方差（`胜可知，而不可为`——有些局本来就输面大）。
2. **防污染**：匿名/陌生反馈只作线索不作依据；改动只由维护者执行；improvement-suggestions 必须附原始案例引用，可追溯。
3. **防漂移**：每次合并前对照 README「伦理边界」和 SKILL.md 原则 6——任何让 skill 更"狠"但越线的建议，直接拒绝并留档。

## 案例沉淀

评分 ≥17（见 scoring-rubric）且用户同意的案例，匿名化后入 examples/：场景标签 + 使用框架 + 策略路径 + 结果 + 点评（好在哪/坑在哪）。examples 同时是新用户的说明书和模型的输出校准器，质量门槛要高，宁缺毋滥。
